#################################知识一（奖励数据的录制技巧）
有，而且你记得的方向基本对，不过官方更常用的说法不是“假成功/假失败数据”，而是：

false positives：明明没成功，分类器却判成成功
false negatives：明明成功了，分类器却没给成功

官方 HIL-SERL 网站先概括说，他们会先遥操作收集正样本和负样本，用来训练一个二分类 reward classifier。

更具体的技巧写在官方 franka_walkthrough.md 里，讲得很明确：

录 reward classifier 数据时，默认所有 transition 都记成负样本，只有你按键标记的那些才记成正样本。
官方明确建议：为了让分类器对 false positives 更稳，负样本最好比正样本多 2 到 3 倍。
而且负样本不是随便拍失败就行，应该专门覆盖容易误判的失败模式，例如：
插在错误位置
只插进去一半
拿着物体但还没真正插入
USB 任务里还特别提到：插进了错误的 USB 口 这种“很像成功但其实是失败”的情况。

这其实就是你说的“假成功/假失败”那个意思，只是官方没有用这个中文词。
更准确地说，官方是在强调：

要专门收集那些“看起来像成功、但实际上是失败”的负样本。

比如文档里就举例说：

如果分类器会把“RAM 条举在空中”误判成成功，那就要多收这种情况的负样本。

另外，官方还说在录 demo 时，如果你发现 reward classifier 出现 false positive 或 false negative，优先做法不是只调阈值，而是继续补针对性的 classifier 数据；必要时甚至增加 classifier 相机或图像。

所以一句话回答你：

有，官方确实强调了这种思路：reward classifier 要靠“成功样本 + 大量覆盖失败模式的负样本”来变稳，尤其要收那些“很像成功但其实失败”的样本，也就是你理解的那种“假成功/假失败”边界数据。

###########################################知识二（训练细节）
任务一旦判成功就结束并复位，那策略学到的到底是什么？部署时怎么连贯？

结论是：

官方这类 HIL-SERL 任务本来就是“按 episode 定义的单次任务”，不是一个无限连续流程。
也就是说，对 USB 插入这种任务，“插成功”本身就是这一回合的终点。demo 录制时，官方明确写的是：一旦 reward classifier 判成功，或者超时，机器人就 reset；录满指定数量的成功 demo 后脚本结束。RAM 插入和 USB pickup/insertion 两个 walkthrough 都是这么写的。

你自己的运行日志也符合这个逻辑：
actor 每个 episode 结束后会打印 [actor-episode-end]，然后立刻进入“正在准备复位 / 开始向底层发送复位坐标 / 单臂平滑复位”的流程。也就是说，你当前代码里，成功或 episode 结束后，后续动作不是“继续做下一个任务步骤”，而是 reset 到下一回合起点。

所以回答你最核心的问题：

1）成功后模型“怎么做”？

官方默认答案是：不继续做别的动作，而是把这一回合视为完成，然后 reset。

因为这类 task 的目标函数就是：

从初始状态出发
完成一次插入
成功即终止

在这种定义下，policy 学的不是“永远不停地工作”，而是：

给定一个新的初始状态，尽快把任务做成功。

这和工业里常见的“单工位循环任务”是一样的。不是策略自己负责决定“下一件事干嘛”，而是外部流程负责把环境恢复到下一个起点，再让策略重新执行一次。

###################知识七（归一化理解）
模型输出一个归一化动作
例如单臂 7 维动作。
先 clip 到 [-1, 1]
防止输出过大。
把前 6 维解释成“这一步的末端位姿增量”
平移增量：action[:3] * POS_SCALE
旋转增量：action[3:6] * ROT_SCALE
把这个增量加到“当前末端位姿”上
也就是：
next_xyz = current_xyz + delta_xyz
next_quat = 当前姿态 + 旋转增量（通过 apply_delta_rotation）
再做安全裁剪
新目标位姿还会过工作空间限制，避免超边界。
最后发送给 ROS / 机械臂
这个“裁剪后的目标末端位姿”才是最终发下去的控制目标。

所以你这句话：

clip再动作缩放后是一次输出的位置增量，再转换成最终的末端位姿吗？

更准确地说是：

先 clip，再缩放得到“本步末端位姿增量”，再加到当前末端位姿上，得到最终目标末端位姿。

再补一个关键点：

平移/旋转 是这样走“增量 -> 目标位姿”的
夹爪 不是这样，它不是连续位姿增量，而是：
-1 -> close
0 -> hold
+1 -> open
然后再映射成硬件量程。

还有一个例外你要记住：

reset 不走这条增量链路。
reset 是直接给绝对目标位姿，不是模型输出动作再 clip/scale。

###########################知识十四（夹爪详细分析）


（1）三值标签：+-1,0分别表示张开，闭合，保持上一状态不动
（2）夹爪惩罚：单独对夹爪的离散dqn奖励计算起作用，不要影响末端的reward
（3）夹爪惩罚处罚条件：夹爪状态反转则处罚，抓物体夹爪状态反转不是很正常，确实正常的，一个任务就几次是合适的，其他时间都是错误的，所以即使正确抓物体也要给惩罚，其他时候惩罚更多，表示不要轻易反转状态。


（4）夹爪三值定义还需要进一步确认准确度和定义的合理性，
demos 和 RLPD 的夹爪保存定义是统一的。
量程反馈用于判断真实夹爪状态；前后状态变化再生成 -1/0/+1 事件标签。
这种方法准确率通常比直接用动作命令高，但要确认 close_max=30 能覆盖“夹住 USB 后”的实际反馈值。
   由于标签准确率高，所以wrapper基于标签给夹爪惩罚的定义也准
    真实夹爪反馈量程
        ↓
    <=30 判定为 closed
    >=70 判定为 open
    30~70 保持上一稳定状态（为了覆盖量程，这个范围几乎不会存在，迅速掠过）
        ↓
    比较 obs -> next_obs 的状态变化
        ↓
    生成 action[6] 事件标签：
     -1 = close event
      0 = hold/no-op
     +1 = open event

这个比直接相信 policy_raw 或 intervene_action 更可靠，因为它记录的是夹爪实际发生了什么，不是“命令想让夹爪做什么”。
（但要加一个前提：夹爪反馈本身可靠，且 CLOSE_MAX=30、OPEN_MIN=70 阈值适合你的真实硬件。(可视化观察闭合后夹爪量程是否仍小于30)）
